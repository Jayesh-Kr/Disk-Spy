### 1.2 Risks identified and how they were handled

1. **MSVC shell must be initialized before `cargo build`.** On a default
   shell, `where link.exe` resolves to `C:\Program Files\Git\usr\bin\link.exe`
   (GNU coreutils), not the MSVC linker. A bare `cargo build` from a normal
   shell would fail.

   **Resolution:** all builds were routed through a wrapper batch file
   (`D:\Disk Spy\build.bat`) that calls `vcvarsall.bat x64` before invoking
   `cargo`. The wrapper is the documented entry point.

2. **No cached `windows` crate.** The cargo registry had no
   `windows-0.58.x` source, and no `Cargo.lock` existed anywhere on the
   system. First `cargo build` would compile the `windows` crate and its
   proc-macro dependency tree from scratch (the build took 3m 30s the first
   time around).

   **Resolution:** confirmed network access to `index.crates.io` worked and
   let the first build populate the cache. Subsequent builds are fast.

## 5. What the user needs to do to use it

DiskSpy requires an **elevated** PowerShell to capture kernel events. From a
non-elevated shell it will print the spec's error message and exit 1. Two
options:

1. **Double-click** `target\release\diskspy.exe` in Explorer — Windows shows
   a UAC dialog, accept it, and DiskSpy runs elevated.
2. **Run as Administrator** from a PowerShell shortcut, then:
   ```powershell
   cd "D:\Disk Spy"
   .\target\release\diskspy.exe
   ```

Once running, open `http://localhost:7272` in any browser. See
`how-to-use.md` for the full walkthrough.

## 6. Spec deviations and the reason

| Spec said | We did | Why |
|---|---|---|
| `ferrisetw = "0.5"` | `ferrisetw = "1.2"` | Version 0.5 was never published on crates.io. The crate jumped from 0.1.1 to 1.0.0 in 2023. The FileIo API is the same. |
| `windows = "0.58"` features list | Same version, **more** features | We needed `Win32_Storage_FileSystem` for `QueryDosDeviceW` and `Win32_System_SystemInformation` for OS version detection, on top of what the spec listed. |
| Build via plain `cargo build` from a normal shell | Build via `build.bat` (wraps `vcvarsall.bat` first) | MSVC linker is not on PATH in a normal Git-Bash / PowerShell. `build.bat` is mandatory on this laptop. |
| UAC elevation via a future code-signed manifest | UAC elevation via `mt.exe` post-build manifest embed | The `embed-manifest` 1.5 API didn't match its docs at the time of the build, so we used the Windows SDK's `mt.exe` instead, called automatically by `build.bat build --release`. |

## 7. Known limitations carried forward

These are inherent to the design and not specific to this laptop:

- ETW kernel tracing needs Administrator. The manifest solves the *launch*,
  not the *driver signing* requirement (none needed — kernel consumer is
  user-mode, no driver).
- Writes inside the WSL2 Linux VM (`\\wsl$\Ubuntu\...` paths) are not
  visible to the Windows kernel tracer.
- ETW reports write bytes, not net file-size change. A "write 100 MB then
  delete" sequence will log `+100 MB`; the actual disk usage never grew.
  Future work: periodic `fs::metadata` reconciliation.
- No first-run baseline scan — DiskSpy only knows about files that change
  after it starts. A `--baseline` flag would address this.
